GPU Kernel Design — ocudu-gpu-channel

How the channel emulator maps wireless-channel math onto the GPU's thread array to emulate the desired signal, interference, and crosstalk for a whole topology in real time.

1. Why a GPU fits channel emulation

A channel emulator turns each transmitter's IQ stream into what a receiver hears. Per output sample the math is independent of every other sample — a gain is a multiply, a phase/CFO rotation is a sincos + complex multiply, AWGN is one RNG draw. That is the textbook shape of a SIMT (single-instruction, multiple-thread) workload: launch one GPU thread per sample and they all run the same chain at once.

OCUDU's ZMQ radio exchanges IQ in 23 040-sample slots (1 ms at 23.04 MS/s). The emulator processes one slot per link per millisecond — so the GPU has a 1 ms budget to shape and sum every edge of the graph.

4. Parallelism axis 2 — each thread sums the whole interference graph

For a node with N incoming edges, the broker stages the N source batches back to back and launches one superpose_kernel. Thread idx walks all N edges for its sample, shaping each through that edge's own model chain, and accumulates — that accumulation is the interference.

Two parallel axes multiply: 23 040 samples wide (one thread each) × N edges deep (the per-thread loop). The 3-node graph's 2-edge nodes are ≈46 000 complex channel evaluations per launch — measured at 5–9 µs on the RTX 5090, well inside the 1 ms slot.

The 3-node graph (topology.graph.cuda.yaml — two UEs and a gNB, uplink interference plus UE↔UE crosstalk) relays through the CUDA broker with every data-integrity counter at zero; the measured node powers match the analytic superposition to <0.3 %. The broker's event=gpu_timings line reports superpose_kernel at 5–9 µs — under 1 % of the 1 ms slot — so the compute headroom scales to many more edges before the GPU is the bottleneck. (The host↔device copies are the larger cost today; overlapping them across the per-node streams is a later optimisation.)
