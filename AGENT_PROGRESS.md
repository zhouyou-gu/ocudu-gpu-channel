## Blockers and Risks

- Remote dependency gap: `iperf3` is still missing on the RTX workstation under the user-space-only constraint.
- Remote GPU driver and CUDA toolkit are visible through `nvidia-smi` and user-space `nvcc`.
- Remote active network path is Wi-Fi (`wlp129s0`), which is suitable for SSH/control but not accepted as a real-time distributed IQ data path.
- Milestone A core path is proven: OCUDU gNB <-> Option B CUDA broker <-> srsUE relays real bidirectional IQ at real time, srsUE reaches `RRC Connected` / `RRC NR reconfiguration successful`, broker counters are all zero, and the gNB internal log has 0 `Real-time failure in RF: overflow`. The earlier broker startup deadlock was a loopback-bind config bug (`ue0.rx_endpoint`), now fixed; the earlier ~512K-1.3M overflow was the free-running synthetic UE tools, not the broker.
- Remaining attach gap: the previous srsUE run logged `Failed to setup/configure GW interface`, so the PDU session was not established and the UE data-plane ping did not run (`attach-summary.json` ended `status=ue_stack_blocker_no_attach`, `pdu_session_established=0`). The script now creates `ue1` before launching srsUE, but the fix still needs a remote attach-smoke rerun.
- `scripts/remote/ocudu-attach-smoke.sh` previously hung on teardown when srsUE wedged on shutdown; fixed to `docker rm -f` the srsUE container before `wait`.
- The 614400-sample CUDA interop benchmark is red under the current slot-duration gate even though the batch spans about 26.7 ms of IQ at 23.04 MS/s; the benchmark gate needs a separate interop-batch interpretation before publishing results.
- The patched OCUDU ZMQ Docker rebuild compiles UHD/OCUDU from source and is slow; a clean rebuild can take many minutes before the gNB reports `zmq` as an available radio type.
- CUDA MVP intentionally does not support AWGN, integer delay, or fractional delay; those model steps fail fast on `backend: cuda`.
- Broker hot path no longer allocates poll vectors or TX receive IQ buffers in each loop, but still constructs link-key strings and uses unordered-map lookups while processing RX requests.
- CUDA MVP still does a `cudaStreamSynchronize` at the end of every `process_into` call, so batches within one link are not pipelined; with `process_mutex` removed the two directions' per-link streams do now overlap, but cross-batch pipelining within a link remains future work.
